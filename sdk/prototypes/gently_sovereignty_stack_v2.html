NixOS ENFORCEMENT
How the Freeze Works
# /etc/nixos/core-policy.nix { gently.core = { # Shell source — read-only for all users shell.source = /nix/store/...-gently-shell; shell.mutable = false; # Core apps — frozen in toolbar pinnedApps = [ "terminal" "git-viz" "gpu-mon" "vault" "network-inspector" ]; pinnedApps.removable = false; # Dev sandbox boundaries sandbox.allowNetwork = true; sandbox.allowGPU = true; sandbox.allowFS = [ "/home/*/projects" ]; sandbox.denyFS = [ "/etc" "/nix" "/boot" ]; }; }
NixOS declarative config enforces the boundaries. The /nix/store is read-only by design. Core apps are installed as system packages — only root (only nix-rebuild, only from your signed flake) can change them. The freeze is filesystem-level, not just UI-level.

TRIPLE SANDBOX
What Feed Apps Can't Do
App from feed runs with TWO sandboxes: SANDBOX 1: OS Level (NixOS enforcement) ✕ Access filesystem outside $APP_DIR ✕ Access network (unless declared + approved) ✕ Modify system config ✕ Escalate privileges SANDBOX 2: Hardlock (5-layer iframe isolation) ✕ Access parent window (Gently Shell) ✕ Run infinite loops (watchdog kills) ✕ Crash without recovery (snapshot rollback) SANDBOX 3: Proton hardlock (if morphable) ✕ App's self-edit can't break the app runtime ✕ Same 5 layers, nested one level deeper
Three sandboxes deep for a morphable feed app. NixOS at the bottom, hardlock in the middle, Proton hardlock at the top. That's the sovereignty guarantee.
